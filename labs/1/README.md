# Лабораторная работа №1

## Перечень заинтересованных лиц (стейкхолдеров)

- **Пользователи системы**  
  Взаимодействуют с системой через Telegram-бота для управления доступом. После получения доступа подключаются к внешним узлам через публичный API с использованием специализированных клиентских приложений.

- **Служба технической поддержки**  
  Использует административные функции системы для просмотра состояния пользователей, доступов и серверов, а также для обработки обращений.

- **Администратор системы**  
  Управляет пользователями, тарифными планами и удалёнными серверами через веб-интерфейс.

- **Владелец продукта**  
  Заинтересован в корректной работе системы, масштабируемости архитектуры и возможности дальнейшего расширения функциональности.

---

## Перечень функциональных требований

- взаимодействие пользователей с системой через Telegram-бота;
- автоматическая регистрация пользователей на основе Telegram-аккаунта;
- управление подпиской и сроками действия доступа;
- автоматическая выдача доступа к удалённым серверам;
- предоставление пользователю параметров доступа для подключения через клиентские приложения;
- автоматическое удаление доступа по истечению срока действия;
- продление доступа при наличии активной подписки;
- приём разовых и рекуррентных платежей через интеграцию с платёжными сервисами;
- управление тарифными планами;
- сбор и хранение статистики использования внешних узлов;
- балансировка пользователей между удалёнными серверами;
- предоставление публичного API для подключения клиентских приложений;
- административное управление пользователями и серверами через веб-интерфейс;
- логирование пользовательских и административных действий.

## Перечень сделанных предположений

- подключение к внешним узлам осуществляется только через публичный API;
- Telegram API и публичный API системы доступны и стабильны;
- платёжные сервисы предоставляют API для интеграции;
- управление внешними узлами осуществляется через программный интерфейс.

---

## Перечень нефункциональных требований

- **Производительность**  
  Публичный API и Telegram-бот должны обеспечивать низкое время отклика при большом количестве одновременных запросов.

- **Масштабируемость**  
  Архитектура должна поддерживать горизонтальное масштабирование Telegram-бота, API и сервисов управления серверами.

- **Отказоустойчивость**  
  При сбоях отдельных компонентов система должна сохранять доступность Telegram-бота и публичного API.

- **Безопасность**  
  Доступ к API должен быть защищён, а пользовательские данные — надёжно изолированы.

- **Доступность**  
  Telegram-бот и публичный API должны быть доступны в режиме 24/7.
